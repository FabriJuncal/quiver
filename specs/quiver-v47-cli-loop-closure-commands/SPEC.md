# Quiver v47 - CLI Loop Closure Commands

**Date:** 2026-05-31
**Status:** Completed
**Source:** User-approved acceptance criteria and production-readiness review derived from `CLI_ANALYSIS.md`.

Slice numbering resets here. This spec intentionally starts at `slice-00`.

## Problem

Some CLI loops remain incomplete or overly deep:

- Users lack a single `status` entrypoint comparable to `git status`.
- `evidence run` can generate evidence but the CLI cannot list or show generated evidence.
- `migrate` lacks a first-class contract changelog surface.
- `demo create spec-viewer` is deeper than necessary for a single available demo.
- `config language` needs an explicit forward contract before simplification.

## Objective

Add or formalize small loop-closure commands without disturbing the v46 compatibility work:

- `status` as a read-only high-level entrypoint.
- `evidence list` and `evidence show`.
- `changelog` and migration guidance.
- `demo spec-viewer` alias while preserving `demo create spec-viewer`.
- A documented decision for the future `config` surface.

## Scope

### Included

- New read-only `status` command.
- `evidence list` and `evidence show <path>`.
- `changelog` command backed by local package/repo content.
- `demo spec-viewer [--dry-run]` as canonical shorthand or alias.
- Explicit config contract decision documented in spec/docs.
- Help, docs, tests, JSON safety where supported, and package-installed smokes.

### Excluded

- Removing `demo create spec-viewer`.
- Downloading remote changelog content.
- Adding new config domains beyond documented contract decisions.
- Parser modernization beyond command additions.

## Approved Acceptance Criteria

1. `status` is read-only and never writes files, starts slices, opens worktrees, runs providers, or prompts.
2. `status` selects the most relevant existing state source and shows a next safe command.
3. `status --json`, if supported, emits parseable JSON with no ANSI or human prose.
4. `evidence list` lists evidence generated by `evidence run` with stable ordering.
5. `evidence show <path>` displays one evidence artifact and rejects unsafe paths.
6. `evidence run` remains compatible with current `-- <command>` usage.
7. `changelog` displays local Quiver contract/version changes without network access.
8. `migrate` help references `changelog` or equivalent migration preview guidance.
9. `demo spec-viewer [--dry-run]` works as a simplified entrypoint.
10. `demo create spec-viewer` remains functional as a compatibility alias.
11. The config command surface decision is documented before any runtime simplification.
12. Help and docs include all new commands and aliases.
13. Tests cover human output, JSON safety where applicable, unsafe paths, no-write behavior, and package-installed smoke.

## Command Contracts

These contracts are the source of truth for implementation slices. They intentionally avoid runtime changes in `slice-00`.

### `status`

- Purpose: top-level read-only summary comparable to `git status` for Quiver workflow state.
- Behavior: selects the most relevant existing local state source in this order: active slice, open AI lifecycle run, approved/blocked planner phase, available next slice, initialized project state, uninitialized project guidance.
- Human output: compact hierarchy with current state, blockers or warnings when present, and exactly one next safe command when one can be determined.
- JSON contract: supported as `status --json` with schema version 1, current state code, source, optional spec/slice/run refs, warnings, blockers, and `next_command`.
- Safety: no file writes, no provider execution, no prompts, no worktree creation, no slice start side effect.
- Compatibility: may reuse existing `flow`, `dashboard`, and resolver helpers, but must not rename their existing JSON fields.

### `evidence list`

- Purpose: browse local evidence generated by `evidence run`.
- Default root: `.quiver/evidence/` under the current project root.
- Ordering: newest first by file mtime, with deterministic path tie-breaker.
- Human output: relative path, timestamp when detectable, size, and first heading/command summary when cheap to read.
- JSON contract: optional; if implemented, emits schema version 1 with an `items` array and no localized prose.
- Safety: read-only, no artifact mutation, no traversal outside the evidence root.

### `evidence show <path>`

- Purpose: display one evidence Markdown artifact.
- Accepted paths: project-relative paths inside `.quiver/evidence/` and absolute paths only when they resolve inside the same evidence root.
- Rejections: traversal, symlinks resolving outside the root, directories, missing files, non-Markdown files unless explicitly documented by the evidence library.
- Human output: artifact content or concise metadata plus content; must not redact differently from the artifact already stored.
- JSON contract: optional; if implemented, emits schema version 1 with path, metadata, and content.
- Compatibility: `evidence run -- <command>` remains unchanged.

### `changelog`

- Purpose: show local Quiver contract/version changes without network access.
- Source: local `CHANGELOG.md` when present; otherwise a packaged fallback summary or a clear no-local-changelog message.
- Human output: latest relevant entries by default, with option-compatible design for future filtering.
- JSON contract: optional; if implemented, emits schema version 1 with entries parsed from local content.
- Safety: no network access, no package-manager calls, no writes.
- Migration link: `migrate` help or guidance must mention `changelog` and `migrate --dry-run` as safe preview paths.

### `demo spec-viewer`

- Purpose: canonical shorthand for the only currently available demo.
- Compatibility: `demo create spec-viewer` remains functional and documented as compatibility form.
- Behavior: equivalent to `demo create spec-viewer` for `--dry-run`, `--dir`, language, no-color, and error handling.
- Safety: dry-run remains no-write; live mode remains scoped to the selected demo target.

### Config Surface Decision

- Runtime decision for v47: keep `config language show|set` as canonical behavior.
- Documentation decision: do not add a broader `config set/get` runtime surface in v47.
- Rationale: the current config surface only owns language precedence and persistence; introducing a generic key/value surface before more config domains exist would create compatibility obligations without user value.
- Documentation impact: command references and UX guidance must describe `config language` as canonical, not temporary.
- Future compatibility requirement: if a simplified config surface is introduced later, it must be additive and preserve `config language` aliases/deprecations with the same stderr-only and JSON-cleanliness rules established in v46.

## v46 Dependencies

- Legacy/deprecation behavior must follow v46: compatibility aliases remain functional, human deprecation warnings go to `stderr`, and JSON-capable flows keep `stdout` parseable.
- New command docs and generated scripts should prefer canonical commands.
- New human errors should use the i18n catalog or be explicitly allowlisted with ownership.
- Read-only commands must include no-write tests when they inspect repository state.

## Guardrails

- Do not remove old commands.
- Keep new commands small and independently testable.
- Do not use network access.
- Keep `status` strictly read-only.
- Do not change `config` behavior until the contract decision is explicit.
